# V2 – Production-Quality Analytics Architecture

## Overview

V2 focuses on evolving the analytics implementation from a simple proof of concept into a more maintainable and production-oriented architecture.

Rather than having pages communicate directly with Google Analytics, analytics responsibilities are separated into dedicated layers with clear responsibilities.

---

# Goals

- Remove direct `window.gtag()` usage from application pages.
- Centralize analytics logic.
- Improve maintainability and scalability.
- Prevent invalid or duplicate conversions.
- Build a foundation for future analytics enhancements.

---

# Milestone 1 – Analytics Refactor ✅

## Architecture

Application pages no longer interact with Google Analytics directly.

Instead, analytics flow through the following layers:

```text
Page
  ↓
Analytics API
  ↓
gtag Wrapper
  ↓
Google Analytics (GA4)
```

---

## Project Structure

```text
src/
│
├── constants
│   └── analytics.ts
│
├── lib
│   ├── analytics.ts
│   └── gtag.ts
│
└── types
    └── analytics.ts
```

### Responsibilities

#### `constants/analytics.ts`

Stores all analytics event names in one place.

Current events:

- `generate_lead`
- `hero_cta_click`

---

#### `types/analytics.ts`

Defines TypeScript interfaces for analytics event parameters.

Current types:

- `GenerateLeadParams`
- `HeroCTAParams`

---

#### `lib/gtag.ts`

Provides a small wrapper around `window.gtag()`.

This is the only file that knows how to communicate with Google Analytics.

---

#### `lib/analytics.ts`

Exposes reusable analytics functions that the rest of the application calls.

Current API:

- `trackGenerateLead()`
- `trackHeroCTAClick()`

Application pages never call `window.gtag()` directly.

---

## Benefits

- Separation of concerns
- Strong typing for analytics events
- Centralized event definitions
- Easier maintenance
- Easier testing
- Scalable architecture for future events

# Milestone 2 – Conversion Validation & Deduplication ✅

## Problem

Initially, visiting `/thank-you` always triggered a `generate_lead` event.

This allowed several invalid scenarios:

- Manual navigation to `/thank-you`
- Duplicate conversions after page refresh
- Repeated conversions during the same session

---

## Solution

A session-based conversion gate was introduced.

### Flow

```text
Landing Page
      │
      ▼
Issue Conversion Ticket
      │
      ▼
CTA Click Event
      │
      ▼
Navigate to /thank-you
      │
      ▼
Is conversion allowed?
      │
 ┌────┴────┐
 │         │
No        Yes
 │         │
 ▼         ▼
Redirect   Track generate_lead
Home            │
                ▼
      Consume Conversion Ticket
```

---

## Session Management

A dedicated session utility centralizes all browser storage logic.

Responsibilities include:

- issuing a conversion ticket
- validating conversion eligibility
- consuming the ticket after a successful conversion

Pages no longer interact with `sessionStorage` directly.

---

## Conversion Behaviour

The application now enforces the following rules:

### Valid flow

```text
Landing Page
      ↓
CTA Click
      ↓
Thank-you Page
      ↓
generate_lead
```

---

### Direct URL Access

```text
/thank-you

↓

Redirect to Home

↓

No conversion recorded
```

---

### Page Refresh

```text
First Visit

↓

generate_lead

↓

Refresh

↓

No additional conversion
```

---

### Back / Forward Navigation

Repeated visits to the confirmation page during the same session no longer generate additional conversion events.

---

## Benefits

- Prevents fake conversions
- Prevents duplicate conversions within a session
- Centralizes session logic
- Improves analytics accuracy
- Better reflects production conversion flows
