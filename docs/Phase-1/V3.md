# Complete Guide: Next.js Google Ads Conversion Tracking System (V3.1)

## Project Overview

This project is a Next.js landing page demonstrating a realistic conversion tracking pipeline.

The system combines:

- Next.js App Router
- React Client Components
- Server Components
- TypeScript
- Google Analytics 4 (`gtag`)
- Browser storage APIs:
  - `sessionStorage`
  - `localStorage`

- Conversion state management
- Event tracking
- Duplicate conversion prevention
- Conversion expiry handling

The goal:

> Track when a user starts a conversion journey, verify completion, send the conversion event to Google Analytics, and prevent invalid or repeated conversions.

---

# 2. Next.js Execution Environment

Next.js runs code in two environments:

1. Server
2. Browser

## Server Environment

The server can:

- Render Server Components
- Generate HTML
- Fetch server data

The server cannot access:

```javascript
window;
document;
localStorage;
sessionStorage;
```

because these APIs only exist inside the browser.

---

## Browser Environment

The browser handles:

- User clicks
- React Client Components
- Browser APIs
- Google Analytics execution

Available APIs:

```javascript
window;
document;
localStorage;
sessionStorage;
```

---

# 3. Server Components vs Client Components

Server Components:

```tsx
export default function Page() {}
```

Used for:

- SEO content
- Static rendering
- Server data fetching

Cannot use:

```tsx
onClick;
useEffect;
useState;
window;
localStorage;
sessionStorage;
```

---

Client Components:

Enabled with:

```tsx
"use client";
```

Required for:

- Click handlers
- Browser APIs
- React effects

The homepage is a Client Component because it contains:

```tsx
<button onClick={handleClick}>
```

A server cannot listen for browser interactions.

# 5. TypeScript Window Declaration

File:

```
src/lib/gtag.ts
```

Contains:

```ts
declare global {
  interface Window {
    gtag?: (...args: unknown[]) => void;
  }
}
```

Purpose:

This only informs TypeScript.

It does not create:

```javascript
window.gtag;
```

The actual function is created by the Google Analytics script.

# 6. Storage Architecture

The application uses two storage systems.

---

# sessionStorage

Purpose:

Stores the active conversion journey.

Example:

```json
{
  "eventId": "abc123",
  "allowed": true,
  "fired": false,
  "createdAt": 1750000000
}
```

---

## eventId

Unique identifier for one conversion.

Used for:

- Debugging
- Tracking
- Future server-side processing

---

## allowed

Determines whether the conversion can continue.

Current implementation:

```ts
allowed: true;
```

---

## fired

Determines whether the conversion event was already sent.

Before:

```json
{
  "fired": false
}
```

After:

```json
{
  "fired": true
}
```

---

## createdAt

Timestamp of conversion creation.

Used for expiry checking.

---

# localStorage

File:

```
src/lib/tracking.ts
```

Purpose:

Stores browser-level CTA tracking state.

Example:

```
hero_click_fired = 1
```

Unlike sessionStorage, localStorage survives:

- Refreshes
- Browser restarts

Current use:

Prevent repeated:

```
hero_cta_click
```

events.

# 7. Conversion Expiry

The conversion now has an expiry period.

Configuration:

```ts
const CONVERSION_EXPIRY_MS = 30 * 60 * 1000;
```

Meaning:

```
30 minutes
```

A conversion older than 30 minutes is considered invalid.

Example:

```
User clicks CTA

↓

Conversion created

↓

User waits 45 minutes

↓

Visits /thank-you

↓

Conversion expired

↓

Redirect home
```

# 9. Session Utility Functions

File:

```
src/lib/session.ts
```

Responsible for conversion lifecycle.

---

## saveConversion()

Stores conversion data.

Flow:

```
Object

↓

JSON.stringify()

↓

sessionStorage
```

---

## readConversion()

Reads stored conversion.

Flow:

```
sessionStorage

↓

JSON string

↓

JSON.parse()

↓

Object
```

---

## getConversion()

Safely exposes conversion data.

It protects against server execution:

```ts
if (typeof window === "undefined")
```

---

## ensureConversion()

Creates a conversion if needed.

Responsibilities:

1. Check existing conversion.
2. Check expiry.
3. Reuse valid conversion.
4. Create a new conversion if expired or missing.

Example:

Existing:

```json
{
  "eventId": "abc",
  "fired": false
}
```

is reused.

Expired:

```json
{
  "eventId":"old",
  "createdAt":old_timestamp
}
```

is replaced.

---

## canConvert()

Checks whether conversion can continue.

It validates:

1. Conversion exists.
2. Conversion is not expired.
3. Conversion is allowed.

---

## consumeConversion()

Marks conversion as completed.

Before:

```json
{
  "fired": false
}
```

After:

```json
{
  "fired": true
}
```

# 13. Analytics Events

The application sends two events.

---

## hero_cta_click

Triggered when the user clicks:

```
Get Started
```

Example:

```ts
trackHeroCTAClick({
  button_name: "Get Started",
});
```

Purpose:

Measure user interest.

It does not represent a completed conversion.

---

## generate_lead

Triggered after successful conversion completion.

Flow:

```
Landing Page

↓

CTA click

↓

Thank-you page

↓

trackGenerateLead()

↓

GA4
```

Example:

```ts
trackGenerateLead({
  lead_source: "landing_page",
  event_id: conversion.eventId,
});
```

Purpose:

Represent a completed lead conversion.

# 17. Current Design Decisions

## Why expiry was added

Without expiry:

A user could:

```
Click CTA

↓

Leave website

↓

Return hours later

↓

Open thank-you URL

↓

Generate old conversion
```

Expiry prevents stale conversions.

---

## Why lead*fired*<eventId> was not added

The project already has:

```ts
conversion.fired;
```

which prevents duplicate generate_lead events.

Additional browser keys are unnecessary at this stage.

---

## Why localStorage extraction was added

Previously:

```
page.tsx

↓

localStorage logic
```

Now:

```
page.tsx

↓

tracking.ts

↓

localStorage
```

Benefits:

- Cleaner component
- Reusable tracking utilities
- Better separation of concerns

# Final Mental Model

The complete system:

```
User

↓

Landing Page

↓

Clicks Get Started

↓

ensureConversion()

↓

sessionStorage saves conversion

↓

hero_cta_click sent

↓

localStorage stores CTA lock

↓

Navigate to /thank-you

↓

Validate conversion

↓

Check expiry

↓

Send generate_lead

↓

consumeConversion()

↓

Google Analytics receives conversion

↓

Google Ads can import it when attribution exists
```

Summary:

- `hero_cta_click` measures interest.
- `generate_lead` measures completed conversion.
- `sessionStorage` manages the active conversion journey.
- `localStorage` prevents repeated CTA tracking.
- `ensureConversion()` creates or preserves valid conversions.
- Conversion expiry prevents stale journeys.
- `gtag()` sends events to Google Analytics.
- Google Ads only attributes conversions when a valid Google Ads interaction exists.

This creates a small but realistic conversion tracking system using Next.js, GA4, and Google Ads.
