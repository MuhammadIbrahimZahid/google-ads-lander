# Complete Guide: Next.js Google Ads Lead Capture System (Phase 2)

## Project Overview

Phase 2 extends the conversion tracking landing page into a complete lead capture system.

Phase 1 focused on:

- Creating conversion journeys
- Tracking CTA interactions
- Sending GA4 events
- Preventing duplicate conversions

Phase 2 adds:

- User lead collection
- API processing
- Server-side validation
- Data sanitization
- PostgreSQL storage
- Campaign attribution tracking

The system now represents a realistic marketing funnel:

```

Google Ad

↓

Landing Page

↓

User Clicks CTA

↓

Conversion Journey Created

↓

User Submits Lead Form

↓

API Validation

↓

Lead Stored in Database

↓

Thank You Page

↓

generate_lead Event

↓

Google Analytics 4

↓

Google Ads Attribution

```

The goal:

> Capture real user information while keeping marketing attribution data connected to the lead record.

# 3. Frontend Lead Form

The lead form is a Client Component.

Reason:

It requires:

```tsx
"use client";
```

because it uses:

- useState
- form events
- browser APIs
- fetch requests

The component manages:

```ts
formData;
```

Example:

```json
{
  "name": "",
  "email": "",
  "phone": ""
}
```

---

# 4. Form Submission Flow

When the user submits:

```tsx
<form onSubmit={handleSubmit}>
```

The application:

1. Prevents browser refresh.

```ts
e.preventDefault();
```

---

2. Creates conversion state.

```ts
ensureConversion();
```

This keeps conversion tracking separate from lead storage.

---

3. Builds lead payload.

Example:

```json
{
  "name": "John Smith",
  "email": "john@test.com",
  "phone": "03000000000",
  "landingPage": "/",
  "referrer": ""
}
```

---

4. Sends API request.

```ts
fetch("/api/leads");
```

---

# 5. API Route Handler

File:

```
src/app/api/leads/route.ts
```

Next.js App Router uses Route Handlers for custom server endpoints. Route Handlers are created using files such as `route.ts` inside the `app` directory and can handle methods like POST requests. ([Next.js][1])

The endpoint:

```
POST /api/leads
```

receives lead submissions.

File:

```

```

- Cleaner API routes
- Easier testing
- Database logic stays isolated

# 20. Phase 2 Design Decisions

## Why use an API route?

The browser should not directly access:

- Database credentials
- SQL queries
- Server resources

The API creates a secure boundary.

---

## Why use a service layer?

Without services:

```
route.ts

↓

SQL
```

With services:

```
route.ts

↓

service

↓

database
```

This keeps responsibilities separated.

---

## Why store attribution data with leads?

Because marketing performance depends on knowing:

```
Which campaign

↓

Generated which lead
```
