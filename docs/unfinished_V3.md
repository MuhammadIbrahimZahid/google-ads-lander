# Complete Guide: Next.js Google Ads Conversion Tracking System

## Project Overview

This project is a Next.js landing page that demonstrates a simplified but realistic conversion tracking pipeline.

The system combines:

- Next.js App Router
- React Client Components
- Server Components
- TypeScript
- Google Analytics 4 (`gtag`)
- Browser APIs:
  - `sessionStorage`
  - `localStorage`
- Conversion state management
- Event tracking
- Duplicate conversion prevention

The goal:

> Track when a user interacts with the landing page, determine whether they completed the conversion journey, send the conversion event to Google Analytics, and prevent counting the same conversion multiple times.

# 2. Next.js Execution Environment

Next.js runs code in two different environments:

1. Server
2. Browser

Understanding this distinction is critical.

---

# Server Environment

The Next.js server:

- Executes Server Components
- Generates HTML
- Fetches server-side data
- Sends HTML to the browser

The server does not have access to:

```javascript
window;
document;
localStorage;
sessionStorage;
```

because these are browser APIs.

---

# Browser Environment

The browser:

- Executes client JavaScript
- Handles user interactions
- Runs React client code
- Provides browser APIs

The browser has:

```javascript
window;
document;
localStorage;
sessionStorage;
```

---

# 3. Server Components vs Client Components

## Server Components

Default in Next.js:

```tsx
export default function Page() {}
```

Run on the server.

Good for:

- SEO content
- Static pages
- Data fetching
- Performance optimization

Cannot use:

```tsx
onClick;
useState;
useEffect;
window;
localStorage;
sessionStorage;
```

---

## Client Components

Enabled using:

```tsx
"use client";
```

Run inside the browser.

Required for:

- Click handlers
- React state
- Effects
- Browser APIs

Example:

```tsx
"use client";

export default function Home() {}
```

Your homepage needs this because it contains:

```tsx
<button onClick={handleClick}>
```

A server cannot listen for browser clicks.

# 9. declare global vs window.gtag

These are different concepts.

---

## TypeScript Declaration

Example:

```ts
declare global {
  interface Window {
    gtag?: Function;
  }
}
```

Purpose:

Only helps TypeScript.

It tells TypeScript:

> The window object may contain a gtag function.

It does not create anything.

---

## Actual Creation

Example:

```javascript
window.gtag = gtag;
```

This creates the real browser function.

Before:

```
window

├── location
├── document
```

After:

```
window

├── location
├── document
├── dataLayer
└── gtag()
```

---

# 10. Why typeof window Exists

Example:

```ts
if (typeof window === "undefined") {
  return;
}
```

Reason:

The server does not have browser APIs.

Without protection:

```ts
window.localStorage;
```

on the server causes:

```
ReferenceError:
window is not defined
```

The check means:

Server:

```
No browser.
Stop.
```

Browser:

```
Browser exists.
Continue.
```

# 11. Storage Architecture

The application uses:

## 1. sessionStorage

Stores the active conversion journey.

Example:

```json
{
  "eventId": "abc123",
  "allowed": true,
  "fired": false,
  "createdAt": 1750000000
}
```

---

## eventId

Unique identifier for a conversion.

Example:

```
abc123
```

---

## allowed

Whether conversion is allowed.

Currently:

```ts
allowed: true;
```

is always assigned.

---

## fired

Whether the conversion was already sent.

Before:

```json
{
  "fired": false
}
```

After:

```json
{
  "fired": true
}
```

---

## createdAt

Timestamp.

Useful for:

- Expiration
- Analytics timing
- Attribution windows

---

# 2. localStorage

Example:

```
hero_click_fired = 1
```

Purpose:

Remember CTA tracking across browser sessions.

It survives:

- Refresh
- Browser restart

---

# 3. lead*fired*<eventId>

Example:

```
lead_fired_abc123
```

Purpose:

Additional duplicate protection.

Meaning:

> This conversion already generated a lead event.

# 14. Session Utility Functions

File:

```
src/lib/session.ts
```

Responsible for conversion state.

---

# saveConversion()

Purpose:

Save conversion object.

Flow:

```
Object

↓

JSON.stringify()

↓

sessionStorage
```

---

# readConversion()

Purpose:

Read conversion object.

Flow:

```
sessionStorage

↓

JSON.parse()

↓

Object
```

Important line:

```ts
JSON.parse(value) as Conversion;
```

Meaning:

1. Convert JSON string into an object.
2. Tell TypeScript:

> Treat this object as a Conversion.

Important:

`as Conversion` does not validate data.

It only helps TypeScript.

---

# getConversion()

Purpose:

Safely expose conversion data.

Protects against server execution.

---

# issueConversion()

Purpose:

Create or preserve a conversion.

It:

1. Reads existing conversion.
2. Creates conversion object.
3. Saves it again.

Why?

To preserve existing data.

Example:

Existing:

```json
{
  "eventId": "abc",
  "fired": true
}
```

Keeps:

```json
{
  "eventId": "abc",
  "fired": true
}
```

Instead of creating:

```json
{
  "eventId": "new",
  "fired": false
}
```

A clearer name would be:

```
ensureConversion()
```

because it ensures a conversion exists.

---

# canConvert()

Checks:

```ts
allowed === true;
```

Meaning:

"Is this conversion allowed?"

---

# markConversionFired()

Changes:

Before:

```json
{
  "fired": false
}
```

After:

```json
{
  "fired": true
}
```

---

# hasFiredConversion()

Checks:

```
Was this conversion already sent?
```

---

# consumeConversion()

Meaning:

Use conversion once.

Like:

```
Coupon:

Unused

↓

Used
```

Conversion:

```
fired:false

↓

fired:true
```

# Final Mental Model

Think of the system as a ticket machine:

```
User clicks CTA

        ↓

Create ticket

        ↓

Store ticket

        ↓

User reaches thank-you page

        ↓

Validate ticket

        ↓

Use ticket

        ↓

Mark ticket consumed

        ↓

Reject reuse
```

The whole system answers:

> "Did this user complete the intended conversion journey, and have we already counted it?"

---

`hero_cta_click` answers:

> "Did the user show interest?"

`generate_lead` answers:

> "Did the user complete the conversion?"

`sessionStorage` manages the current visit.

`localStorage` remembers browser-level history.

`gtag` sends events to Google Analytics.

The application is a small **conversion tracking state machine built on top of Next.js**.
